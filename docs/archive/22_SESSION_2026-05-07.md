## 1. Search Bug Fix — "WP/4068/2025 returns no results"

### The Problem

Users searching for `WP/4068/2025` got no results. The old search code did:
```sql
WHERE cc.case_ref ILIKE '%WP/4068/2025%'
```

This never matches because Calcutta HC stores the case as `WPA/4068/2025` — the type is **WPA** (Writ Petition Appellate), not **WP**. The user typed a prefix ("WP") expecting to match all writ petitions, but the ILIKE on the full string requires an exact substring match.

### The Fix

Parse the search input into three parts — type prefix, case number, year — and match each separately.

**File:** `src/eventtrace/storage/repositories/causelist.py`

Added regex at top of file:
```python
_SEARCH_CASE_REF_RE = re.compile(
    r"^([A-Za-z][A-Za-z0-9\.\(\)\-]*)[\s/]+(\d+)(?:[\s/]+(\d{4}))?$"
)
```

When a case_ref search arrives, the code now tries to parse it:
- `WP/4068/2025` → type=`WP`, number=`4068`, year=`2025`
- Query becomes: `case_type ILIKE 'WP%' AND case_number = '4068' AND case_year = 2025`
- This matches `WPA/4068/2025`, `WP.CT/4068/2025`, `WP.ST/4068/2025`, etc.

If the input doesn't match the pattern (e.g. free-text like "paul vs state"), it falls back to the old ILIKE on the full case_ref string.

Both the SQLite and Postgres backends were updated with the same logic.

## 3. Matters System

A "matter" is a case that a lawyer has formally registered in EventTrace for billing purposes. This is the foundation of the billing system.

### 3a. Database

**File:** `src/eventtrace/storage/postgres.py`

New table added to `ensure_schema()`:
```sql
CREATE TABLE IF NOT EXISTS matter (
  id                 SERIAL PRIMARY KEY,
  user_id            UUID NOT NULL,
  case_ref           TEXT NOT NULL,
  case_title         TEXT,
  case_type          TEXT,
  case_number        TEXT,
  case_year          INTEGER,
  court_no           TEXT,
  petitioner         TEXT,
  respondent         TEXT,
  status             TEXT NOT NULL DEFAULT 'active',
  billing_mode       TEXT NOT NULL DEFAULT 'appearance',
  fee_per_appearance NUMERIC(12,2),
  notes              TEXT,
  opened_at          DATE,
  closed_at          DATE,
  created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(user_id, case_ref)
)
```

Five repository methods added: `list_matters`, `get_matter`, `create_matter`, `update_matter`, `delete_matter`.

`create_matter` uses `INSERT ... ON CONFLICT (user_id, case_ref) DO UPDATE ... RETURNING *` — so re-registering the same case just updates it rather than erroring.

`update_matter` uses a dynamic `SET` clause built at runtime — only the fields you pass get updated, using an explicit allowlist to prevent injection.

## 4. eCourts CAPTCHA Solver

### Why We Need It

The eCourts case lookup form at `hcservices.ecourts.gov.in` requires solving a CAPTCHA before returning case data. We use this to auto-fill matter details (petitioner name, respondent name, court number) so lawyers don't have to type everything manually.

### Why Not Playwright

Playwright is a browser automation tool. It launches a full Chrome/Chromium browser (~400MB binary, 3–5 second cold start). We do **not** need it here because:

- The eCourts form is **plain PHP** — no JavaScript rendering, no dynamic content
- Submitting the form is a standard HTTP POST to `case_no_qry.php`
- The response is raw text (tilde-delimited), not HTML that needs rendering
- The only "stateful" part is keeping the `PHPSESSID` cookie across 3 requests — `requests.Session` handles this trivially

Using `requests.Session` instead: 50KB library, instant startup, no flakiness.

### How It Works

**File:** `src/eventtrace/services/ecourts.py`

The flow (from the docstring):
1. `GET case_no.php` → server sets `PHPSESSID` cookie in our session
2. `GET securimage_show.php` → fetches CAPTCHA image bytes (same session = same CAPTCHA answer stored server-side)
3. Send image to **Claude Haiku vision** → get back 5-char alphanumeric text
4. `POST case_no_qry.php?action_code=showRecords` → server validates CAPTCHA + returns results
5. Parse the tilde-delimited response
6. Retry up to 3 times if CAPTCHA is wrong (server returns `error1`)

### CAPTCHA Solving with Claude

```python
def _solve_captcha_with_claude(image_bytes: bytes, api_key: str) -> str:
    client = anthropic.Anthropic(api_key=api_key)
    msg = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=20,
        messages=[{
            "role": "user",
            "content": [
                {"type": "image", "source": {"type": "base64", "media_type": "image/png", "data": b64}},
                {"type": "text", "text": "This is a CAPTCHA image. It contains exactly 5 alphanumeric characters. Reply with ONLY those characters, nothing else."},
            ],
        }],
    )
    return msg.content[0].text.strip()
```

Cost: ~$0.000025 per solve (Haiku pricing). Success rate ~80% per attempt; with 3 retries, effective success rate is ~99%.

### Parsing the Response

eCourts returns tilde-delimited records, each terminated by `##`:
```
{internal_id}~{case_ref}~{parties_html}~{cino}~{court_no}~...##
```

Example:
```
12345~WPA/4068/2025~RUPA PAUL AND ANR.<br/>Versus<br/>STATE OF WB~CALMA...~3~##
```

The parser (`_parse_result`):
1. Splits on `##` to get individual records
2. Splits each on `~`
3. Extracts `case_ref` from field[1] via regex
4. Strips HTML tags from field[2], splits on "versus" to get petitioner / respondent
5. Takes court_no from field[4]

### CASE_TYPE_IDS

eCourts requires a numeric type ID in the POST payload, not the abbreviation. We maintain a dict of 75 entries mapping abbreviation → ID:
```python
CASE_TYPE_IDS = {
    "WPA": 12, "FMA": 19, "CRM": 17, "CO": 15, ...
}
```

If the case type abbreviation isn't in this dict, we raise `ValueError` immediately (no HTTP request made). This is important because Original Side types (CO, WPO) use a different court_code endpoint — currently not supported, planned for Step 9.

Added `ANTHROPIC_API_KEY` to settings (`src/eventtrace/core/config.py`). The ecourts-lookup endpoint returns HTTP 503 if the key is not configured. Added `"anthropic>=0.40.0"` and `"requests>=2.31"` to `pyproject.toml` dependencies.

## 5. URL Routing Fix

### The Problem

Before this session, all pages in the frontend loaded at the same URL. Navigating to Causelist, Search, Matters, etc. all showed the same URL in the browser. This means:
- No bookmarking — you can't bookmark the Search page or share a direct link to Matters
- No back/forward — browser back button doesn't take you to the previous page
- On reload, always lands on Live Board regardless of where you were

### The Fix

**File:** `EventTrace-Web/src/App.tsx`

Hash-based routing: each page gets a URL fragment (`#board`, `#causelist`, `#search`, `#my-cases`, `#matters`, `#profile`).

Zero new dependencies — uses the browser's native `window.location.hash` and `hashchange` event.

```typescript
function getPageFromHash(): Page {
  const hash = window.location.hash.replace('#', '') as Page
  return VALID_PAGES.includes(hash) ? hash : 'board'
}
```

```typescript
// In App component
const [page, setPage] = useState<Page>(getPageFromHash)  // reads hash on first load

useEffect(() => {
  function onHashChange() { setPage(getPageFromHash()) }
  window.addEventListener('hashchange', onHashChange)
  return () => window.removeEventListener('hashchange', onHashChange)
}, [])  // back/forward navigation

function navigate(p: Page) {
  window.location.hash = p  // updates URL
  setPage(p)
}
```

Nav links changed from `<button onClick>` to `<a href="#page" onClick={e => { e.preventDefault(); navigate(n.id) }}>` — so right-click → "Copy link address" gives a shareable URL, and middle-click opens in new tab.

Why not react-router-dom? For a single-page app with 6 static pages and no dynamic URL params, hash routing is ~50 lines of code vs adding a 50KB dependency and restructuring the whole component tree. Hash routing also works on Vercel/Railway without any server-side config (no need to rewrite all URLs to `index.html`).
